## 1. Цель первой версии (MVP)

Рабочая система для **индивидуальных** и **групповых** дыхательных практик с синхронизацией через сервер.

### Входит в MVP

| Компонент | Функционал |
|-----------|------------|
| Telegram Bot | Команды: `/start`, `/create`, `/join`, `/launch`, `/pause`, `/resume`, `/stop`, `/status`, `/history`, `/stats` |
| Web App | Визуальный квадрат, анимация фаз, таймер, прогресс |
| Режимы дыхания | Фиксированный квадрат, прогрессивный квадрат |
| Ограничения | По общему времени **или** по количеству проходов |
| Групповая синхронизация | Комнаты, единый серверный таймлайн |
| Статистика | Участники онлайн, завершившие, % прогресса, базовая история |

### Не входит в MVP (отложено)

| Компонент | Версия |
|-----------|--------|
| Режимы: Пирамида, Обратная пирамида, Волна | v2 |
| Административная панель (п. 13 ТЗ) | v2 |
| Звук, фоны, социальные функции (п. 16 ТЗ) | v3+ |
| Публичные комнаты, расписание, подписки | v3+ |

## 2. Архитектура

```
┌─────────────────┐     ┌─────────────────┐
│  Telegram Bot   │────▶│    Backend      │
│  (aiogram)      │     │  FastAPI + WS   │
└─────────────────┘     └────────┬────────┘
                                 │
┌─────────────────┐              │
│  Web App        │◀─────────────┤
│  React + TS     │   WebSocket   │
└─────────────────┘              │
                                 ▼
                    ┌────────────────────────┐
                    │  PostgreSQL  │  Redis  │
                    └────────────────────────┘
```

### Роли компонентов

- **Telegram Bot** — создание сессий, управление, кнопки, уведомления в группах
- **Web App** — визуализация дыхания, подключение участников, отображение прогресса
- **Backend** — хранение настроек, комнаты, синхронизация времени, статистика
- **Redis** — состояние активных комнат, pub/sub для real-time
- **PostgreSQL** — комнаты, сессии, участники, история

### Стек

| Слой | Технологии |
|------|------------|
| Backend | Python 3.12+, FastAPI, SQLAlchemy, PostgreSQL, Redis, WebSocket |
| Bot | aiogram 3.x, Telegram Bot API |
| Frontend | React, TypeScript, Vite |
| Инфра | Docker, docker-compose |

## 4. Этапы разработки

### Этап 0 — Подготовка

**Задачи:**
- [ ] Создать структуру каталогов `backend/`, `frontend/`
- [ ] Настроить `docker-compose.yml` (PostgreSQL, Redis)
- [ ] Добавить `.env.example` с переменными окружения
- [ ] Зарегистрировать бота у [@BotFather](https://t.me/BotFather)
- [ ] Настроить HTTPS-туннель для разработки (ngrok / Cloudflare Tunnel)

**Результат:** локальное окружение поднимается одной командой.

---

### Этап 1 — Ядро дыхания (без Telegram)

**Задачи:**
- [ ] Модель фаз: вдох → задержка после вдоха → выдох → задержка после выдоха
- [ ] Режим «Фиксированный»: параметры не меняются (например 4-4-4-4)
- [ ] Режим «Прогрессивный»: рост от начального до конечного значения с шагом и числом проходов
- [ ] Ограничение по времени: автоматический расчёт числа циклов
- [ ] Ограничение по проходам: явное задание количества проходов
- [ ] Unit-тесты на генерацию расписания циклов

**Результат:** модуль `breathing` с предсказуемым API, покрытый тестами.

**Пример API модуля:**
```python
schedule = BreathingEngine.create(
    mode="progressive",
    inhale=4, hold_in=4, exhale=4, hold_out=4,
    start_value=4, end_value=20, step=1, passes=3,
    duration_limit_minutes=30,  # или passes_limit=3
)
# schedule.phases → список фаз с длительностью и метаданными
```

---

### Этап 2 — Backend API + WebSocket

**Задачи:**
- [ ] Модели БД: Room, Session, Participant, SessionStats
- [ ] REST API:
  - `POST /rooms` — создать комнату
  - `GET /rooms/{id}` — получить настройки
  - `POST /rooms/{id}/join` — присоединиться
  - `POST /rooms/{id}/start` — запустить
  - `POST /rooms/{id}/pause` / `resume` / `stop`
  - `GET /rooms/{id}/status` — статус и участники
- [ ] WebSocket `/ws/rooms/{id}` — синхронизация состояния
- [ ] Сервер как единственный источник времени (server timestamp + phase index)
- [ ] Redis для активных комнат и broadcast событий
- [ ] Валидация Telegram `initData` для Web App

**Результат:** комната создаётся, участники синхронизируются через WebSocket.

**Синхронизация:**
- Сервер хранит `started_at`, текущую фазу и индекс цикла
- Клиент получает состояние и интерполирует локально
- Целевое рассогласование UI: **≤ 300 мс** (допустимо для мобильных сетей)
- Серверная логика фаз: **≤ 100 мс** точности

---

### Этап 3 — Web App (визуал)

**Задачи:**
- [ ] Компонент `BreathingSquare`: 4 стороны, подсветка активной фазы
- [ ] Анимация индикатора по периметру квадрата
- [ ] Отображение: фаза, осталось секунд, текущий уровень, проход, общий прогресс, осталось времени
- [ ] Подключение к WebSocket, реконнект при обрыве
- [ ] Адаптивная вёрстка (мобильные + десктоп)
- [ ] Интеграция Telegram Web App SDK (`Telegram.WebApp`)

**Результат:** Web App открывается в браузере/туннеле, показывает синхронную практику.

**Макет экрана практики:**
```

│      Название практики       │
│                              │
│         ┌─────────┐          │

│         │  ВЫДОХ  │          │
│         └─────────┘          │
│                              │

```

### Этап 4 — Telegram Bot

**Задачи:**
- [ ] `/start` — приветствие, краткая инструкция
- [ ] `/create` — мастер создания практики (inline-кнопки для параметров)
- [ ] `/join` — кнопка «Присоединиться» → Web App URL
- [ ] `/launch` — запуск практики в группе
- [ ] `/pause`, `/resume`, `/stop` — управление сессией
- [ ] `/status` — текущее состояние комнаты
- [ ] `/history`, `/stats` — базовая статистика
- [ ] Публикация кнопки «Присоединиться» всем участникам группы

**Результат:** полный сценарий «админ создаёт → участники присоединяются → синхронная сессия».

**Сценарии:**

| Сценарий | Поток |
|----------|-------|
| Групповой | Админ `/create` → `/launch` → кнопка «Присоединиться» → Web App → синхрон |
| Соло | Пользователь `/create` в личке → сразу Web App → практика без комнаты |

---

### Этап 5 — Статистика и полировка

**Задачи:**
- [ ] Сохранение: дата, время начала, длительность, режим, параметры
- [ ] По комнате: участники, среднее время, завершившие
- [ ] Отображение в боте: `/history`, `/stats`
- [ ] Обработка edge cases: отключение участника, истечение времени, досрочный stop
- [ ] Smoke-тест: 10–50 одновременных подключений

**Результат:** MVP готов к демонстрации и тестированию в реальной группе.

## 5. Модель данных (черновик)

### Room
| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| title | string | Название практики |
| description | string? | Описание |
| owner_id | bigint | Telegram user ID |
| chat_id | bigint? | ID группы (null для соло) |
| breathing_config | JSON | Параметры дыхания |
| status | enum | draft / waiting / active / paused / finished |
| created_at | datetime | Дата создания |

### Session
| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| room_id | UUID | FK → Room |
| started_at | datetime? | Время старта |
| ended_at | datetime? | Время окончания |
| duration_sec | int | Фактическая длительность |

### Participant
| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| session_id | UUID | FK → Session |
| user_id | bigint | Telegram user ID |
| joined_at | datetime | Время подключения |
| finished_at | datetime? | Время завершения |
| completed | bool | Завершил ли практику |

## 7. Корректировки относительно TZ.md

| Пункт ТЗ | Решение для MVP |

| 5 режимов дыхания (п. 4) | Только фиксированный + прогрессивный |
